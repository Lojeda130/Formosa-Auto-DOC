2 Procedimientos — Onboarding de Nuevo Tester
📋 Paso a Paso — Primer Día
Objetivo Que un tester nuevo pueda ejecutar su primer test en menos de 2 horas desde cero, entendiendo la estructura del proyecto y el flujo de trabajo. Esta guía asume una PC limpia con Windows.

🔑 Fase 0 — Solicitar accesos (antes de arrancar)

0
Solicitar al líder técnico o responsable
  • Acceso al repositorio en GitLab/GitHub (rol mínimo: Developer)
  • Acceso a la instancia de Jenkins (para visualizar builds y reportes)
  • Acceso a la red interna / VPN si los servicios del banco no son accesibles desde fuera
⚠️ Sin estos accesos el onboarding no puede completarse. Gestionar esto el día anterior o a primera hora.

🔧 Fase 1 — Instalación del ambiente base

1
Instalar JDK 21

Descargar JDK 21 (21.0.x) LTS desde cualquier proveedor: jdk.java.net, Amazon Corretto, Eclipse Temurin u Oracle JDK. Instalar y luego configurar las variables de entorno manualmente:

Windows — configurar variables de entorno del sistema:

Panel de Control → Sistema → Configuración avanzada del sistema → Variables de entorno → Variables del sistema

  • Crear nueva variable: JAVA_HOME = ruta de instalación, ej: C:\Program Files\Java\jdk-21
  • Editar variable Path → agregar nueva entrada: %JAVA_HOME%in
💡 Abrir una terminal nueva después de guardar. Las terminales ya abiertas no cargan las variables recién configuradas. 
java -version    // openjdk version "21.0.x"
echo %JAVA_HOME% // Debe mostrar la ruta del JDK
2
Instalar Maven 3.9

Descargar el binario (zip) desde maven.apache.org/download.cgi. Descomprimir en una ruta fija, por ejemplo C: ools\maven-3.9. Luego configurar variables de entorno del sistema:

  • Crear nueva variable: MAVEN_HOME = C: ools\maven-3.9
  • Editar variable Path → agregar nueva entrada: %MAVEN_HOME%in
💡 Abrir una terminal nueva antes de verificar — las variables recién configuradas solo se cargan en terminales nuevas. 
mvn -version // Debe mostrar: Apache Maven 3.9.x, Java version: 21
⚠️ Si mvn -version muestra otra versión de Java: Verificar que JAVA_HOME apunte al JDK 21 y que la entrada %JAVA_HOME%in aparezca antes que cualquier otro JDK/JRE en el Path.
3
Instalar Git y configurar identidad

Descargar desde git-scm.com. Durante la instalación seleccionar "Git from the command line and also from 3rd-party software". Una vez instalado:

git config --global user.name  "Nombre Apellido"
git config --global user.email "[email protected]"
git --version  // git version 2.4x

Configurar acceso SSH al repositorio:

# Generar clave SSH (aceptar ruta por defecto, passphrase opcional)
ssh-keygen -t ed25519 -C "[email protected]"

# Copiar la clave pública
type %USERPROFILE%\.ssh\id_ed25519.pub

Pegar esa clave pública en GitLab/GitHub → Perfil → SSH Keys. Verificar la conexión:

ssh -T git@[servidor-gitlab]  // Welcome to GitLab, @usuario!
4
Instalar Node.js 20 LTS

Requerido para Allure CLI y, si se trabaja con mobile o desktop, para Appium. Descargar el instalador LTS desde nodejs.org. El instalador configura el Path automáticamente.

node -v   // v20.x.x
npm -v    // 10.x.x

Agregar variable de entorno del sistema:

  • Crear NODE_PATH = %AppData% pm ode_modules
  • Verificar que %AppData% pm esté en el Path (el instalador suele agregarlo)
5
Instalar Allure CLI 
npm install -g allure-commandline
allure --version  // 2.25.x
💡 Alternativa sin Node.js: Descargar el binario directamente desde github.com/allure-framework/allure2/releases, descomprimir y agregar la carpeta bin/ al Path.
6
Instalar un IDE

El tester puede usar el IDE de su preferencia. Las dos opciones más habituales para Java:

IntelliJ IDEA

Opción popular para proyectos Java/Maven. La edición Community es suficiente.

  • Descargar desde jetbrains.com/idea
  • Al abrir: File → Open → carpeta raíz del proyecto
  • Configurar JDK 21 (21.0.x): File → Project Structure → SDKs → Add JDK
  • IntelliJ detecta el pom.xml padre y descarga dependencias automáticamente
  • Si no detecta Maven: Maven → Reload All Maven Projects
Visual Studio Code

Alternativa liviana. Requiere extensiones adicionales.

  • Descargar desde code.visualstudio.com
  • Instalar: Extension Pack for Java (incluye soporte Maven y debug)
  • En settings: "java.jdt.ls.java.home": "C:\...\jdk-21"
  • Abrir carpeta raíz: File → Open Folder
  • Si Maven no importa automáticamente: abrir la paleta de comandos (Ctrl+Shift+P) → Java: Import Java Projects into Workspace
✓
Verificación completa del ambiente — abrir una terminal nueva antes de ejecutar 
java -version    // openjdk version "21.0.x"
mvn -version     // Apache Maven 3.9.x — Java version: 21
git --version    // git version 2.4x
node -v          // v20.x.x
allure --version // 2.25.x
echo %JAVA_HOME%  // ruta del JDK 21
echo %MAVEN_HOME% // ruta de Maven
echo %NODE_PATH%  // %AppData%
pm
ode_modules
✅ Todos deben responder correctamente antes de continuar. Si alguno falla revisar variables de entorno y abrir terminal nueva (las variables se cargan al abrir la terminal).

📱 Fase 1b — Instalación adicional (solo módulos mobile / desktop)

Solo necesario para testers que trabajen en los módulos mobile o desktop. Los testers que trabajen únicamente con web o api pueden omitir esta fase.
1b.1
Instalar Appium Server 2.x 
npm install -g appium
appium --version  // 2.x.x
1b.2
Instalar drivers de Appium según módulo 
# Para mobile Android
appium driver install uiautomator2
# Para mobile iOS (solo en agente macOS)
appium driver install xcuitest
# Para desktop Windows
appium driver install windows

# Verificar instalados
appium driver list --installed
1b.3
[Mobile] Instalar Android Studio y configurar emulador (AVD)

Descargar desde developer.android.com/studio. Durante la instalación incluir el Android SDK. Luego:

  • Abrir Virtual Device Manager y crear un AVD con Android 12+
  • Crear variable ANDROID_HOME = %LocalAppData%\Android\Sdk
  • Agregar al Path: %ANDROID_HOME%\platform-tools y %ANDROID_HOME%\emulator
adb devices  // List of devices attached (con el emulador corriendo)
1b.4
[Mobile iOS] Configurar Xcode y iOS Simulator (solo en Mac)

iOS solo puede automatizarse desde un agente macOS. Si se trabaja localmente en iOS:

  • Instalar Xcode 15+ desde la App Store de macOS
  • Instalar Xcode Command Line Tools: xcode-select --install
  • Abrir Xcode → Settings → Platforms → descargar el simulador iOS deseado (iOS 16+)
  • Instalar Node.js 20 LTS en la Mac y ejecutar appium driver install xcuitest
xcodebuild -version  // Xcode 15.x
xcrun simctl list    // Lista los simuladores disponibles
appium driver list --installed  // xcuitest debe aparecer
⚠️ Esta configuración no aplica en la VM Windows. El agente macOS debe estar registrado como nodo Jenkins para que los builds de iOS corran automáticamente en CI.
1b.5
[Mobile] Instalar Appium Inspector

Descargar desde github.com/appium/appium-inspector/releases. Herramienta visual para identificar los localizadores (selectores) de elementos en la app antes de escribir los tests.

1b.6
[Desktop] Instalar WinAppDriver 1.2.1

Descargar desde github.com/microsoft/WinAppDriver/releases (versión 1.2.1). Instalar con permisos de administrador.

Habilitar Developer Mode en Windows (obligatorio):

Configuración → Actualización y seguridad → Para desarrolladores → activar "Modo desarrollador"

⚠️ Sin Developer Mode activo WinAppDriver no puede iniciar sesión Appium. Reiniciar el equipo luego de habilitar esta opción.
1b.7
[Desktop] Instalar Accessibility Insights for Windows

Descargar desde accessibilityinsights.io. Herramienta visual para identificar los localizadores de elementos en aplicaciones de escritorio Windows antes de escribir los tests.

📥 Fase 2 — Obtener el código

7
Clonar el repositorio 
# Clonar usando SSH — la URL exacta la provee el referente técnico
git clone git@[servidor]:[grupo]/automation-framework.git

cd automation-framework

# Ver todas las ramas disponibles (locales y remotas)
git branch -a
8
Posicionarse en la rama dev de la célula asignada 
# Ejemplo: célula "productos"
git checkout dev-celula-a

# Si la rama no existe localmente todavía:
git checkout -b dev-celula-a origin/dev-celula-a

git status  // Confirmar rama activa y estado limpio
9
Abrir el proyecto en el IDE y descargar dependencias

Abrir la carpeta raíz desde el IDE. El IDE detecta el pom.xml padre automáticamente. Si no lo detecta, forzarlo manualmente desde el panel de Maven.

# Descargar todas las dependencias de todos los módulos
# (primera vez puede tardar 5-10 min según la conexión)
mvn dependency:resolve -q
💡 Primera descarga: Maven descarga todos los JARs del repositorio remoto. Solo ocurre la primera vez; las siguientes ejecuciones usan la caché local en %USERPROFILE%\.m2 epository.
10
Verificar que todos los módulos compilan sin errores 
mvn clean compile -DskipTests  // BUILD SUCCESS en todos los módulos
⚠️ Si falla el BUILD:
  • Verificar que JAVA_HOME apunta al JDK 21
  • Ejecutar mvn clean install -DskipTests para reinstalar módulos locales
  • Revisar conectividad al repositorio Maven (puede requerir proxy/VPN)
11
Verificar environments.yml

El archivo core/src/main/resources/environments.yml está versionado en el repositorio privado. Al clonar el proyecto ya estará disponible con las URLs y credenciales de cada ambiente configuradas. Verificar que el ambiente test esté completo antes de ejecutar:

# Estructura del archivo:
environments:
  test:
    baseUrl: https://[url-ambiente-test]
    apiUrl:  https://[url-api-test]
    timeout: 30
  prep:
    baseUrl: https://[url-ambiente-prep]
💡 El archivo ya viene completo con el clone. Solo verificar que los valores del ambiente test sean correctos. Si algún valor está vacío o desactualizado, consultarlo con el referente técnico.

🛠️ Fase 2b — Configurar ejecución desde el IDE

Opcional pero recomendado. Permite ejecutar y depurar tests individuales con un click desde el IDE sin usar la terminal. A continuación se describen los pasos para IntelliJ IDEA y VS Code, las dos opciones más habituales.
2b.1
IntelliJ IDEA — Configurar Run/Debug Configuration

Ir a Run → Edit Configurations → + → JUnit y completar:

Campos a configurar

Nameej: API - test
Moduleseleccionar api (o el módulo a ejecutar)
Test kindClass para una clase, Method para un método
VM options-Denv=test -Dplatform=web -Dheadless=false

Uso

  • Hacer click derecho sobre una clase o método de test → Run o Debug
  • En modo Debug se pueden poner breakpoints y ver el estado de las variables en tiempo real
  • Guardar varias configuraciones (una por módulo/ambiente) para cambiar rápido
2b.2
Visual Studio Code — Configurar launch.json

Abrir la paleta de comandos (Ctrl+Shift+P) → Java: Debug Current File, o crear manualmente .vscode/launch.json en la raíz del proyecto:

{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "java",
      "name": "Debug Test - API",
      "request": "launch",
      "mainClass": "",
      "vmArgs": "-Denv=test -Dplatform=web -Dheadless=false",
      "projectName": "api"
    }
  ]
}

Con esto, hacer click derecho sobre un método de test → Run Test o Debug Test aplica los parámetros automáticamente.

🧪 Fase 3 — Ejecutar el primer test

12
Ejecutar el módulo API (punto de entrada recomendado — no requiere drivers ni emuladores) 
# Ejecutar todos los tests del módulo api
mvn test -pl api -am -Denv=test

# Ejecutar una clase específica
mvn test -pl api -am -Denv=test -Dtest=NombreDeClaseTest

# Ejecutar un método específico
mvn test -pl api -am -Denv=test -Dtest="NombreDeClaseTest#nombreDelMetodo"
✅ ¿Qué esperar? La consola muestra la ejecución de JUnit. Al final aparece BUILD SUCCESS (o BUILD FAILURE si un test falla — lo cual es válido, significa que el framework funciona correctamente).
13
Ver el reporte Allure localmente 
# Genera y abre el reporte en el browser automáticamente
allure serve api/target/allure-results

Se abrirá el browser con el reporte HTML interactivo: pasos de ejecución, parámetros enviados, respuestas recibidas y screenshots si los hay.

14
[Opcional] Ejecutar módulo web con Playwright 
mvn test -pl web -am -Denv=test -Dplatform=web -Dheadless=false
# Con headless=false se ve el browser abrirse y ejecutar las acciones
💡 Primera ejecución: Playwright descarga los browsers necesarios (~300MB). Solo ocurre una vez.
15
[Opcional] Ejecutar módulo mobile con Appium

Seleccioná la variante según cómo vas a ejecutar los tests:

🤖 Android — Emulador (AVD) click para expandir

Requiere tener Android Studio instalado y un AVD creado (ver Fase 1b).

# Paso 1: iniciar el emulador desde Android Studio
# Tools → Device Manager → ▶ Play en el AVD deseado
# Esperar hasta que el emulador termine de bootear

# Paso 2: (nueva terminal) iniciar Appium Server
appium
// Debe mostrar: Appium REST http interface listener started on 0.0.0.0:4723

# Paso 3: (nueva terminal) ejecutar los tests
mvn test -pl mobile -am -Denv=test -Dplatform=android
✅ Verificación previa: adb devices debe mostrar el emulador como emulator-5554 device antes de lanzar Appium.
📱 Android — Dispositivo físico click para expandir

Requiere habilitar el modo desarrollador en el dispositivo y conectarlo por USB.

En el dispositivo Android:

  • Ir a Ajustes → Acerca del teléfono → tocar Número de compilación 7 veces
  • Ir a Ajustes → Opciones de desarrollador → activar Depuración USB
  • Conectar el cable USB y aceptar el diálogo de autorización en el dispositivo
# Verificar que el dispositivo es reconocido
adb devices
// Debe mostrar: [serial] device (no "unauthorized")

# Obtener el nombre del dispositivo para las capabilities
adb shell getprop ro.product.model

# Iniciar Appium
appium

# Ejecutar (el framework detecta automáticamente el dispositivo conectado)
mvn test -pl mobile -am -Denv=test -Dplatform=android
💡 Capabilities de dispositivo real: El serial del dispositivo se configura en environments.yml bajo la clave deviceName del ambiente correspondiente.
🍎 iOS — Simulador (requiere Mac) click para expandir

Solo ejecutable en macOS. Requiere Xcode y el driver xcuitest instalado (ver Fase 1b).

# Paso 1: listar simuladores disponibles
xcrun simctl list devices available

# Paso 2: bootear el simulador deseado
xcrun simctl boot "iPhone 14"
// O abrirlo desde Xcode: Window → Devices and Simulators

# Paso 3: iniciar Appium
appium

# Paso 4: ejecutar los tests
mvn test -pl mobile -am -Denv=test -Dplatform=ios
⚠️ Solo en agente macOS en CI. En Jenkins, estos tests se ejecutan en el nodo agente-ios. Para desarrollo local, correr desde una Mac con el simulador activo.
📱 iOS — Dispositivo físico (requiere Mac + certificado) click para expandir

Requiere macOS, Xcode, y un certificado de desarrollador Apple para instalar la app en el dispositivo.

Prerequisitos:

  • Cuenta de Apple Developer gratuita (suficiente para desarrollo local y Simulador). No se requiere cuenta paga — la propuesta asume ejecución en Simulador para CI, no en dispositivos físicos de distribución.
  • La app debe estar firmada con el certificado correcto — configurar en Xcode Signing & Capabilities
  • Conectar el dispositivo iOS por USB y confiar en la computadora desde el dispositivo
  • Habilitar en el dispositivo: Ajustes → Privacidad y seguridad → Modo Desarrollador
# Verificar que Xcode reconoce el dispositivo
xcrun xctrace list devices
// El dispositivo físico debe aparecer en la lista

# Instalar WebDriverAgent (requerido por xcuitest) la primera vez
// Se hace automáticamente al ejecutar el primer test si Xcode está configurado

# Ejecutar con UDID del dispositivo
mvn test -pl mobile -am -Denv=test -Dplatform=ios   -DdeviceUdid=$(idevice_id -l | head -1)
⚠️ Más complejo que el Simulador. Para CI se recomienda usar el Simulador. El dispositivo físico es útil para validar comportamientos específicos de hardware (cámara, biometría, GPS) que el Simulador no puede reproducir.

Fase 4 — Primer ciclo de trabajo con Git

16
Actualizar la rama dev de la célula antes de crear subrama 
git checkout dev-celula-a
git pull origin dev-celula-a  // Siempre antes de crear subrama nueva
17
Crear la subrama de trabajo para la tarea asignada 
# Convención: dev-[célula]-[tipo]-[descripción]
git checkout -b dev-celula-a-feat-flujo-negocio    // nueva funcionalidad
git checkout -b dev-celula-a-fix-timeout-login     // corrección de bug
git checkout -b dev-celula-a-test-simulacion       // test de regresión
18
Desarrollar y commitear con mensajes descriptivos (Conventional Commits) 
git add .
git commit -m "feat(web): agrega test de flujo principal de negocio

- Implementa FlujoNegocioTest con flujo completo
- Agrega locators en AppSelectors
- Cubre caso base y caso de borde"
19
Pushear y abrir Merge Request hacia la rama dev de la célula 
git push origin dev-celula-a-feat-flujo-negocio

En GitLab/GitHub: crear Merge Request apuntando a dev-celula-a. Asignar a un revisor del equipo y esperar el code review.

20
Merge a la rama de célula (protegida) — solo con autorización del líder

Una vez validados los cambios en dev-celula-a, el líder hace el merge hacia la rama celula-a (rama estable protegida). Jenkins detecta el cambio y ejecuta automáticamente el pipeline del módulo correspondiente.

✅ ¡Ciclo completo! El tester nunca toca directamente la rama de célula. Solo trabaja en subramas y dev. Los pushes a la rama estable son siempre gestionados por el líder técnico.
