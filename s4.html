¿Cuándo extraer una constante?

  • El valor aparece en más de un método o clase
  • Es un timeout (puede necesitar ajuste)
  • Es un texto de UI que puede cambiar
  • Es un dato de test reutilizable
  • Tiene significado de negocio claro
Regla multi-módulo Si una constante se usa en más de un módulo Maven, debe subir al módulo core.

Ejemplo correcto

// SistemaWorkflow.java
public final class SistemaWorkflow {
  private SistemaWorkflow() {}

  public static final String
    STEP_INGRESO_DATOS = "step1",
    STEP_SIMULACION   = "step2",
    STEP_EVALUACION   = "step3";
}
✅ Buenas Prácticas de Testing
Principios FIRST para tests
1
Rápidos

Preferir mocks o datos de BD pre-cargados cuando sea posible. Los tests lentos desalientan la ejecución frecuente.

2
Independientes

Cada test debe poder correr solo, sin depender del resultado de otro. No compartir estado entre tests.

3
Repetibles

El mismo test debe dar el mismo resultado cada vez. Evitar dependencia de datos manuales o estado externo no controlado.

4
Autoverificables

Usar assertions explícitas (assertTrue, assertEquals). El test debe fallar con un mensaje claro si hay un problema.

Page Object Model — Reglas
  • Un Page/Screen Object por pantalla. No mezclar lógica de varias pantallas en una clase.
  • Fluent Interface: los métodos de acción retornan this o el siguiente Page/Screen para encadenamiento.
  • Sin assertions en Page Objects: los Page Objects no hacen asserts. Las verificaciones van en la clase de test.
  • Locators como variables de instancia: declarar todos los locators al inicio de la clase, no dentro de los métodos.
  • Métodos compuestos para flujos frecuentes: loginAs(user, pass) en lugar de llamar fill + click por separado en cada test.
Anti-patrón a evitar No copiar-pegar bloques de código entre tests. Si el mismo flujo aparece en dos tests, extraerlo a un método en el Page Object o a un método @BeforeAll de setup.
Etiquetado obligatorio con @Tag

Todos los tests deben tener al menos un @Tag para permitir ejecuciones selectivas en Jenkins:

TagCriterio para usarlo
@Tag("smoke")Test crítico, flujo principal, ~5 min ejecución total
@Tag("regression")Cobertura amplia, corre en cada merge a rama de célula
@Tag("flujo")Automatización de flujos de negocio (crear préstamo, usuario, etc.)
@Tag("api")Tests exclusivamente de capa API
@Tag("wip")Work In Progress — excluir de CI hasta completar
Checklist de Code Review — Automatización

Clases y estructura

  • Page Objects Web usan sufijo Page
  • Screens Mobile/Desktop usan sufijo Screen
  • Tests usan sufijo Test
  • Utils/Helpers con sufijo correcto
  • No hay más de 3 niveles de package

Tests

  • Método usa prefijo should
  • Tiene @DisplayName descriptivo
  • Tiene al menos un @Tag
  • No depende de otro test
  • No tiene datos hardcodeados inline

Page Objects & Locators

  • Locators son variables de instancia
  • Locators tienen prefijo correcto (btn, txt…)
  • Métodos retornan this o siguiente Page
  • Sin asserts dentro del Page Object
  • Constantes en clases Constants dedicadas
